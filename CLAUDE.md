## First-Time Setup

**Run `/setup` before using any other skill.** The setup skill collects your profile, work experience, and Gmail credentials, then generates:
- `personal-docs/master_profile.md` — your full experience doc (source of truth)
- `personal-docs/job_tracker.md` — application tracker
- `personal-docs/user.json` — contact info and config (gitignored)
- `tools/content.py` — your resume bullet library
- `.env` — Gmail credentials for sending

After setup, use `/generate-resume`, `/apply`, `/send-email`, or `/scan-linkedin`.

## Fast Generation — Use the Template

`tools/template.py` is a pre-wired scaffold. Read `tools/content.py` to discover your available role functions and bullet keys.

**Steps:**
1. Read the JD and `personal-docs/master_profile.md`
2. Read `tools/content.py` to discover available role functions and bullet keys
3. Copy `tools/template.py` → `output/applied/[Company] - [Role] - [YYYY-MM-DD]/generate.py`
4. Fill in: `company`, `role`, `date`, `dept`, `accent`, `summary`, `skills`, `experience` bullet keys, `cover`
5. Run `python3 "output/applied/[Company] - [Role] - [YYYY-MM-DD]/generate.py"` from the project root, then delete generate.py
6. Update `personal-docs/job_tracker.md`

**Bullet balance per role — default to 13 bullets total for a 2-page resume:**
- Most recent role: 6 bullets (always use full complement)
- Second-most-recent: 4 bullets
- Earlier roles: 2 bullets each
- Oldest role: 1 bullet

Total: 13 bullets → produces a clean 2-page resume. Reduce only if the JD scope is very narrow.

**Skills section:** trim to 6-7 rows max. Only include categories relevant to the JD — drop the rest.

## Workflow — Generating a Tailored Application

1. Read the JD — extract required skills, preferred skills, tone, keywords, must-haves
2. Read `personal-docs/master_profile.md`
3. Build a keyword map (see Keyword Mirroring Strategy below)
4. Flag anything in the JD not covered by the profile before generating
5. Create folder: `output/applied/[Company] - [Role] - [YYYY-MM-DD]/`
6. Generate into that folder:
   - `[Name]_[Company].docx` — tailored, ATS-friendly
   - Only generate email, cover letter, or pitch doc if explicitly requested
7. Re-read the resume after generation: keyword coverage, bullet quality, length, formatting
8. Update `personal-docs/job_tracker.md`
9. Flag if role states "no sponsorship" — check user's sponsorship situation in `personal-docs/user.json`

## ATS Systems — How They Work

ATS (Applicant Tracking Systems) like Workday, Greenhouse, Lever, iCIMS, Taleo parse resumes before a human ever sees them. Optimize for the parser first, human reader second.

**What ATS parsers do:**
- Extract text and score it against job description keywords — low score = auto-rejected
- Parse sections by header labels — wrong labels cause data to be misclassified
- Fail on: tables, text boxes, headers/footers, multi-column layouts, images, special characters, fancy fonts

**Keyword matching rules:**
- Use the exact phrasing from the job description, not synonyms — "event-driven architecture" not "message-based system"
- Include both spelled-out and abbreviated forms where relevant: "Amazon Web Services (AWS)"
- Hard skills, tools, and technologies are the highest-weight keywords — match these first
- Soft skills matter less for ATS but include if explicitly in the brief
- Repeat important keywords naturally across multiple bullets — frequency signals relevance

**Section headers ATS recognizes:**
- Work Experience / Experience / Professional Experience
- Education
- Skills / Technical Skills
- Projects / Key Projects
- Summary / Professional Summary
- Do NOT use creative labels like "What I've Built" — parsers won't recognize them

**ATS-safe formatting:**
- Standard fonts: Arial, Calibri, Georgia, Times New Roman — 10–12pt body, 14–16pt name
- No tables, text boxes, or multi-column layouts
- No headers/footers for contact info — put it in the body
- No icons, graphics, logos, or decorative borders
- Bullet character: solid bullet (•) or hyphen (-) — no custom symbols
- File format: .docx is safest; most modern ATS also accept PDF
- Dates: consistent format — "Jul 2024 – Present" or "2024 – Present"

## Resume Rules

### Structure (in order)
1. **Header** — Name, phone, email, LinkedIn, portfolio, city (no full address)
2. **Summary** — 2–3 sentences, keyword-dense, tailored to the specific role
3. **Technical Skills** — grouped by category, inline, comma-separated
4. **Experience** — reverse chronological; company, title, dates, location, bullets
5. **Personal Projects** — do not include on senior-level resumes; experience carries sufficient weight. Only for junior/mid roles where breadth needs demonstration.
6. **Education** — degree, institution, year (no GPA)

> Certifications section: only include if `personal-docs/user.json` has a non-empty `certifications` array. Never add it speculatively.

### Bullet Point Formula

> **[Strong action verb] + [what you did / built / solved] + [result / metric / scope]**

Examples:
- "Scaled reactive messaging pipeline from 400K to 1M messages/day on a single instance by restructuring transaction boundaries and adding query timeouts"
- "Eliminated recurring DB connection pool exhaustion (50 connections, 3-day outage) by auditing reactive chain transaction scope and enforcing write-only transaction blocks"

**Action verbs by category:**
- Built / Engineered / Developed / Implemented / Designed
- Scaled / Optimized / Reduced / Eliminated / Improved
- Led / Presented / Drove / Aligned / Collaborated
- Debugged / Diagnosed / Investigated / Resolved / Restored
- Migrated / Integrated / Deployed / Automated / Configured

### Quantification Rules
Always ask: how many? how fast? how much? before/after? how often?
- Volume: "400K messages/day", "50 DB connections", "6 state machines"
- Scale: "single instance", "across 3 services"
- Impact: "2x throughput", "3-day outage eliminated", "zero data loss"
- Scope: "8-month engagement", "4 production dashboards", "2 sprints"

### When There Are No Numbers

Never use vague adjective outcomes ("significantly improving", "markedly enhancing"). Use one of these instead:

**1. End with what it enabled (functional outcome):**
- Weak: "Integrated Kafka for real-time messaging, improving system reliability"
- Strong: "Integrated Kafka for real-time messaging to synchronize software updates across distributed store systems"

**2. End with what it eliminated or prevented (binary outcomes don't need numbers):**
- Weak: "Fixed payload formatting issues, improving data consistency"
- Strong: "Resolved payload formatting bugs causing downstream systems to silently reject incoming data, restoring correct processing across all workflows"

**3. Use scope/scale as the weight-carrier:**
- Weak: "Managed CI/CD pipelines, ensuring smooth deployments"
- Strong: "Managed Kubernetes and Jenkins CI/CD pipelines ensuring high availability across 5,000+ stores"

**4. Drop the outcome entirely — let a strong verb + context carry it:**
- Weak: "Built biometric authentication system, greatly improving reliability"
- Strong: "Built biometric authentication system with end-to-end encryption: C++ DLLs integrated into Java via JNI, SourceAFIS fingerprint matching, device-agnostic across all employee verification points"

**Rule:** If you can't end with a real number or a concrete functional outcome, cut the trailing clause. A clean strong bullet beats one padded with adjective soup.

### Tense
- Current role: present tense ("Owns", "Processes", "Designs")
- Past roles: past tense ("Built", "Migrated", "Resolved")
- No "I" — implied first person throughout

### Length
- 2 pages: standard for engineers with 4+ years across multiple distinct roles
- Never pad — every bullet must carry weight
- Never go to 2 pages just to add white space

### Formatting Rules for Output Docs
- No em dashes in bullet or email content — use commas, colons, or restructure the sentence
- No hyphens (-) in messages and emails — use commas, colons, or restructure the sentence
- Arrows in migration bullets ("Azure to AWS") written as "to" not "→" for ATS safety
- All .docx files generated via python-docx (Bash tool) — never markdown

## Summary Section — How to Write It

The summary is prime ATS real estate:
- 2–3 sentences max
- Front-load the exact job title they're hiring for
- Include 3–5 hard skills or technologies from the brief
- State one strong differentiator (scale, ownership, a specific domain)

Example:
> "Senior Backend Engineer with 5+ years building high-throughput distributed systems in Java. Specialized in reactive, event-driven architectures on AWS (Kinesis, SQS, IAM), with production ownership of pipelines processing 1M+ messages/day. Track record of diagnosing complex system failures and delivering hardened, observable services."

Tailor this for every single role — never reuse verbatim.

---

## Technical Skills Section — How to Format It

Group by category, inline, comma-separated:

```
Languages:        Java (primary), TypeScript, Python
Frameworks:       Spring Boot, Hibernate / JPA, Reactor (reactive)
Cloud:            AWS (Kinesis, SQS, Secrets Manager, IAM, S3, CI/CD)
Messaging:        AWS Kinesis, AWS SQS, Apache Kafka, DLQ patterns
Databases:        MongoDB, MySQL, PostgreSQL, Amazon RDS
Observability:    New Relic (APM, dashboards, alerting), Splunk
DevOps:           Kubernetes, Docker, Jenkins, Git
Patterns:         Microservices, event-driven architecture, reactive / non-blocking
```

Adjust categories and terms based on what the brief emphasizes and what the user actually knows. Drop irrelevant rows.

## Application Email — How to Write It

**Structure (2–3 paragraphs):**
1. **Who you are + proof points** — current role, 1–2 strongest credentials with real numbers, direct relevance to the JD.
2. **Gaps / location / honest notes** — flag any stack gaps or location situation briefly. Skip if nothing to flag.
3. **Work auth + close** — use the `work_auth_para` from `personal-docs/user.json`. End with "Resume attached, available for a call."

**Tone:**
- Confident and direct — write like a senior engineer, not a job seeker
- No: "I am excited to apply", "I believe I would be a great fit", "Thank you for your consideration"
- Yes: "The work aligns directly with...", "Here's what I'd bring to this problem"
- 150–250 words total

## Keyword Mirroring Strategy

Before writing, build a keyword map from the brief:

1. **Must-match** — tools/languages/platforms listed as required: match these exactly
2. **High-value** — tools listed as preferred or mentioned multiple times: include if truthful
3. **Role language** — how they describe the work ("event-driven", "platform engineering"): use their words
4. **Soft/cultural signals** — "ownership", "fast-paced", "cross-functional": weave into bullets or summary

Never force a keyword where it's untrue. Never stuff — some ATS now penalize it and humans notice.

## LinkedIn Job Scanner — /scan-linkedin

Run `/scan-linkedin` to browse LinkedIn search results, evaluate posts against your requirements, and append qualified ones to `jobs.txt`.

**Setup required:**
- `.env` must have `LI_EMAIL` and `LI_PASSWORD` (LinkedIn credentials)
- `personal-docs/job_requirements.md` must exist and reflect your stack, experience, and work auth
- `.mcp.json` is included — it configures the Playwright MCP server (`@playwright/mcp@latest` via npx). No setup needed.

**How it works:**
1. Reads config from `.env` (queries, days back, max posts)
2. Logs into LinkedIn via Playwright browser
3. Runs each search query, extracts post cards, evaluates each against `job_requirements.md`
4. Saves qualified posts to `jobs.txt` in a structured format
5. Reports a summary of what was saved, skipped, and why

**Key files:**
- `.env` — LinkedIn credentials and search config (gitignored)
- `.env.example` — template to copy and fill in
- `personal-docs/job_requirements.md` — your matching rules (update before first use)
- `jobs.txt` — output: curated list of qualified posts
